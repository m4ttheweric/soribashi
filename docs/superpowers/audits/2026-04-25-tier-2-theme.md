## File 2: `compose-theme.ts`

### Mantine analog: `merge-mantine-theme/merge-mantine-theme.ts` + `merge-theme-overrides/merge-theme-overrides.ts`

**Mantine's merge approach:**
- `mergeMantineTheme(currentTheme, themeOverride)`: does a `deepMerge(currentTheme, themeOverride)`, then applies a special rule for `fontFamily` → `headings.fontFamily` propagation, then validates primaryColor/primaryShade. No public `extends` keyword; consumers call `mergeThemeOverrides(overrideA, overrideB)` to combine overrides before passing to the provider.
- `mergeThemeOverrides(...overrides)`: `overrides.reduce((acc, o) => deepMerge(acc, o), {})`.

### Decision-point enumeration

| # | Mantine behavior | Soribashi behavior | Classification |
|---|---|---|---|
| CM-1 | `deepMerge(currentTheme, themeOverride)` — deep-merges the entire object tree recursively | `mergeTokens()` — hand-written per-family merge; not a generic deep merge | `INTENTIONAL` — soribashi's schema-aware merge gives explicit control over each field's merge strategy |
| CM-2 | Colors are a tuple (10+ fixed-length array); merging replaces the array (deepMerge) | Colors are a `Record<string, ColorScale>` (open key set); merging is done per shade key via `mergeNamedScales` | `INTENTIONAL` — soribashi's open color scale design vs Mantine's fixed-tuple design |
| CM-3 | `validateMantineTheme` — throws if primaryColor/primaryShade invalid | No validation | `INTENTIONAL` — soribashi has no `primaryColor`/`primaryShade` concept; intent-based system instead |
| CM-4 | `fontFamily` propagates to `headings.fontFamily` automatically during merge | No cross-field propagation | `INTENTIONAL` — soribashi uses separate `fontFamily.heading` token; no propagation needed |
| CM-5 | `mergeThemeOverrides` is a variadic utility for combining multiple overrides | No equivalent public utility (overrides handled via `extends` chain) | `INTENTIONAL` — soribashi's `extends` design covers the same use case |
| CM-6 | Merges `components` via `deepMerge` — individual component config keys are deep-merged | Merges `components` via `{ ...base.components, ...(child.components ?? {}) }` — child REPLACES entire component entry | `INTENTIONAL` — soribashi's choice: per-component config replacement is simpler and less surprising than deep-merge of nested config keys |
| CM-7 (BUG) | N/A (Mantine's `ThemeTokens` analog includes all fields) | `mergeTokens()` merged only `colors/radius/spacing/fontSize/fontFamily/shadow/breakpoint/zIndex` — missing `fontWeight`, `lineHeight`, `heading` | `BUG` — **FIXED** (see below) |

### BUG found and fixed: CT-BUG-1 through CT-BUG-5

**Bug description:** `mergeTokens()` in `compose-theme.ts` was not merging three `ThemeTokens` fields — `fontWeight`, `lineHeight`, and `heading` — despite these fields being present in the `ThemeTokens` interface. When a child theme extended a base theme that set any of these fields without re-declaring them, the values were silently dropped.

This affected both the `tokens` merge path and the `dark` token merge path (which also flows through `mergeTokens`).

**Root cause:** Fields were added to `ThemeTokens` (as part of the blocks adaptation — `fontWeight`, `lineHeight`, `fontFamily.heading`, and `heading.sizes` for `Title`) but `mergeTokens()` was not updated to handle them.

**Fix:** Added `fontWeight`, `lineHeight`, and `heading` to `mergeTokens()`. For `heading`, a dedicated `mergeHeadingTokens()` helper handles the nested `sizes` (shallow merge by heading level) and `textWrap` (child overrides base).

**Tests added (failing before fix, passing after):**
- `CT-BUG-1`: child inherits base fontWeight tokens through composeTheme
- `CT-BUG-2`: child can override individual fontWeight keys via composeTheme
- `CT-BUG-3`: child inherits base lineHeight tokens through composeTheme
- `CT-BUG-4`: child inherits base heading tokens through composeTheme
- `CT-BUG-5`: composeTheme preserves fontWeight when dark tokens have fontWeight

**Mantine source reference:** The bug is in soribashi-only logic (no Mantine analog for the exact `composeTheme` pattern). The comparison point is Mantine's `deepMerge` which handles all fields generically — soribashi's schema-aware merge must be kept up to date with the schema.
