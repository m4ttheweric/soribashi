## Findings

### P1 — Focus ring is invisible on transparent Button variants

**File:** `apps/pilot/src/recipes/Button/Button.css:47`

The Button recipe sets:

```css
.cr-Button-root:focus-visible {
  outline: 2px solid var(--cr-button-bg);
  outline-offset: 2px;
}
```

For `outline`, `ghost`, and `link`, each intent block sets `--cr-button-bg: transparent`, so keyboard focus gets a transparent custom outline. Because the custom `outline` declaration replaces the UA default focus outline, those variants lose visible keyboard focus.

This also contradicts the journal statement that Wave 1 leaves the browser default `:focus-visible` ring intact on transparent variants.

**Suggested fix:** route focus through a dedicated `--cr-button-focus-ring` variable, falling back to border or text color for transparent variants. Add a Playwright focus-visible assertion for at least one transparent variant.

### P2 — Button drops forwarded refs

**File:** `apps/pilot/src/recipes/Button/Button.tsx:50`

`definePolymorphicComponent` provides `ref` in the render context, but Button ignores it:

```tsx
render: ({ Element, props, getStyles }) => {
```

The rendered `<Element>` never receives `ref={ref}`, so consumers cannot focus, measure, or integrate this Button with ref-based libraries. This is especially important for buttons and anchors because focus management is a common consumer use case.

**Suggested fix:** destructure `ref` from the render context and pass it to `<Element ref={ref} ...>`. Add unit tests for both the default `<button>` path and the polymorphic `as="a"` path.
